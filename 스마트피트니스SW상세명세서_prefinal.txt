## 주제: 스마트 피트니스 관리 시스템
## Project Background 가이드
- Project에 대한 배경 지식을 설명한다.
- Project에서 개발하고자 하는 제품의 필요성/역할, 사용자, 사용자에게 제공할 핵심 기능과 가치/기대효과를 기술한다.
- 시장 환경, 기존/경쟁 제품 분석 및 비교 등을 간략하게 설명한다
## 작성
본 프로젝트는 24시간 무인 운영 피트니스 시스템을 개발을 목표로 한다.
최근 최저 임금 상승과 야간 근무 수당 증가로 인해 24시간 무인 운영 시스템이 크게 확산되고 있다. 또한 팬데믹 이후 직원과의 접촉을 최소화하는 비대면 소비문화가 일상화되면서 무인 점포에 대한 선호도가 높아졌다. 궁극적으로 무인화는 인력 배치 없이 24시간 운영을 가능하게 하여, 심야 시간대의 잠재 고객 수요에 효과적으로 대응하고 새로운 수익을 창출하는 경제성을 제공한다.

이런 거시적인 변화는 피트니스 산업에 가장 빠르게 적용되어, 2025년 기준 전국 헬스장 약 30%가 24시간 운영 형태로 급격히 확산되고 있다. 이 중 상당수는 야간에 무인으로 운영되는 ‘셀프 피트니스’ 형태이다. 그러나 대부분의 현행 시스템은 단순 출입 통제에만 머물러 있고, 청결 관리, 시설 유지보수, 그리고 심야 시간대 보안 문제 등 무인 운영의 핵심 과제들을 해결하지 못하고 있다. 이는 결국 운영 비효율과 고객 불만족을 야기하고 있다.

본 프로젝트는 기존 시장의 명확한 한계점을 극복하고 지능형 무인 운영을 구현함으로써 고객 편의성 증대 및 수익 확대를 목표로 한다. 'AI 기반 지능형 통합 운영 플랫폼'구현을 통해, 고객은 안면 인식 기반의 통합 보안 시스템으로 여러 지점을 자유롭게 이용할 수 있다. 또한,  AI가 헬퍼의 작업 사진을 분석하여 관리자의 수동 개입을 최소화하는 운영 완전 자동화를 실현합니다. 최종적으로 이 시스템은 고객, 지점주, 헬퍼를 유기적으로 연결하여 데이터 기반의 서비스 품질 향상이라는 선순환 구조를 완성한다.

❖ **24시간 무인**으로 운영되는 피트니스 시설 체인 관리
- 고객은 한번의 스마트 피트니스 시스템 등록을 통해 **여러 지점을 자유롭게** 이용 가능
- **안면 인식을 통한 편리한 출입 관리** 및 접근 편이성 증대를 통한 피트니스 수익 확대
- **헬퍼의 작업 사진 자동 인증**을 통한 피트니스 운영 효율화
- **자연어 질의**를 통해 기존 고객들의 리뷰 및 지점 소개글에 기반한 피트니스 지점 검색

## 구축할 시스템 서버
스마트 피트니스 관리 시스템 서버
- 연계대상은 다음과 같이 7개이다. 
	1. 사용자 앱
	2. 헬퍼 앱
	3. 관리자 앱
	4. 신용카드 검증 시스템 -> 신원 인증 서비스 제공
	5. 상용 LLM 서비스 (AI 시스템) -> 자연어 처리 분석 위주
	6. 지점 설비 시스템 -> 카메라 촬영, 스마트 게이트 개방
## 시스템 주요 기능
==고객앱==
- 고객 계정 등록
	- 고객 앱을 통해 계정을 생성. <- precondition
	- 신용카드를 이용해서 본인 인증을 수행하고, 계정명과 암호를 등록함. <- 본인 인증, 계정 암호 등록  
	- 피트니스 출입을 위한 안면 사진을 촬영하며, 서버에 업로드 함. <- 안면 사진 저장
- 고객 리뷰 등록
	- 고객은 피트니스 이용 후 고객 앱을 통해 텍스트 리뷰를 남길 수 있음.
- 지점 검색 with 자연어 검색 <- 기존 고객의 리뷰 내용과 지점 소개를 검색에 포함시켜야 함
	- 고객 앱을 통해 자연어 형식으로 피트니스 지점 검색을 할 수 있는데, 
	- 이때 기존 고객들의 리뷰 내용과 지점주의 지점소개를 포함하여 검색에 활용해야 함.

==지점==
- 안면인식 출입 게이트 개방 요청 <- 고객/헬퍼의 등록된 안면인식인지판독후 결과 리턴
	- 사진을 서버로 전달하면, 서버는 사진을 분석하고 등록된 고객으로 판정하면,  서버가 스피드게이트에 신호를 보내서 게이트가 열리게 됨.
- 고장 점검 수행 -> extend 고장인 경우 관리자 앱을통해 지점주에게 통보 
	- 시스템은 카메라 장치나 스피드게이트 등에 대한 고장 점검을 수행하며, 
	- 고장이라고 판단되면 관리자 앱을 통해 해당 피트니스 지점주에게 통보해 줌

==관리자앱==
- 지점주 계정 등록 <- 안면 사진 등록 안함!!!!!!
	- 관리자 앱을 통해 계정을 생성. <- precondition
- 지점 소개 등록 <- precondition 로그인
	- 지점주는 관리자 앱을 통해 로그인하고, 지점 정보 (위치, 규모 등)를 포함하여 지점에 대한 소개를  텍스트로 입력할 수 있음. 
- 지점 리뷰 조회 
	- 해당 지점에 남겨진 고객의 리뷰를 조회할 수 있음.
- 헬퍼 세탁물 판독 조회 <- 사진과 판독 결과 표시 -> extend 세탁물 판독 결과 수정 -> extend 세탁물 판독 컨펌 -> include 양호인 경우 헬퍼의 잔고 증가
	- 지점주는 관리자 앱을 통해 업로드된 세탁물 사진들을 조회할 수 있음.  이때 각 사진 옆에는 시스템이 자동 분석한 결과가 함께 표시됨. 
	  extend 처리 -> 지점주는 특정한 사진을 선택하여 결과를 오버라이드 할 수 있고, 컨펌할 수 있음.
	- 컨펌된 사진 중 양호로 판정된 건들을 집계하여 헬퍼의 잔고를 증가시킴.

==헬퍼앱==
- 헬퍼 계정 등록
	- 헬퍼 앱을 통해 계정을 생성. <- precondition
	- 피트니스 출입을 위한 안면 사진을 촬영하며, 서버에 업로드 함. <- 안면 사진 저장
- 세탁물 정리 촬영 등록 <- 일 3회 기준여부 체크, 세탁물 사진 업로드
	- 헬퍼는 피트니스의 세탁물 관리를 담당하며, 업무를 마치면 정리된 세탁물 사진을 촬영하고 시스템에 업로드함. 
	  지점별로 일 3회 세탁물을 정리한다고 가정함.
- 헬퍼 세탁물 판독 처리 -> include LLM 서비스 활용
	- 시스템은 업로드된 세탁물 사진을 분석하고, 각 사진에 대해 적절하게 정리되어 있는지 분석하여 
	- 그 결과를 양호/미흡/불분명으로 구분하여 저장함.

==시스템 핵심 차별화 기능==
- MLOps 안면인식/세탁물 학습을 통한 오판독률 개선
- 고객이 검색한 자연어 내용을 바탕으로 고객 성향 데이터를 생성, 해당 성향에 맞는 지점 소개나 리뷰글 등록 시 해당 성향을 가진 고객들에게 알림 전송


1.2 Besiness Context Diagram
**System Context의 Multiplicity 명세**
- **지점(매장) 수:** 전국 기준 100개 매장(지점) 목표로 시스템을 기획/구축
    - 각 매장은 독립적으로 운영되는 실매장(지점)이며, 각 매장당 1명의 지점주(Branch Owner)가 배정됨
- **지점주(Branch Owner):**
    - 지점당 1명 × 100개 지점 = **총 100명**
    - 각 지점의 운영·관리 책임자로 시스템 내에서 독립적 권한 행사
- **헬퍼(Helper):**
    - 매장별 평균 10명 내외의 헬퍼가 실무(세탁물 작업 등) 담당
    - 100개 매장 × 평균 10명 = **약 1,000명**
    - 실제 헬퍼는 특정 지점에 소속되어 작업하며, 작업 인증/사진 판독/보상 적립 등 기능 이용
- **고객(Customer):**
    - 전국 100개 매장의 전체 활성 회원 수 기준으로 시스템을 설계
    - 매장별 평균 100명 고객 × 100개 지점 = **약 10,000명**
    - 일반 소비자/회원 고객이 안면인식 출입, 검색/리뷰, 알림 이용 등 시스템의 주요 기능을 사용

### 로그인

System Interface: HTTPS (JSON/REST API)
[역할]

- 보안: SSL/TLS, 패스워드 해시 저장

- 입력 데이터: 자연어 텍스트 (최대 100자 내외)
- 출력 데이터: 지점 리스트 (JSON)
- 빈도: 일 평균 2,000건 총 고객 10,000명 중 일일 활성 사용자(DAU) 20% 1회 검색 지군), 피크 타임 시: 시간당 500건 (일 트래픽의 50%가 18~20시 2시간에 집중)

[유형]

[역할]

- 설비는 이 요청에 즉시 응답하며, 일정 시간 내 응답이 없으면 시스템은 설비 고장 알림을 발신한다.
[특성]
- 입력 데이터: 설비 ID 
- 빈도: 설비 상태/고장 보고 실패율에 의존적(일 14,400건 보고 중 1% 실패/누락 가정 시, 일 144건)

[특성]
- 데이터: 알림 메시지 (Text)
- 빈도: 고장 발생 시 (비정기적, 낮음)

- 설명: 시스템에 안면 인식을 위한 사진을 전송하여 출입 인증을 요청하고, 주기적인 설비 상태를 보고하는 하드웨어입니다. 시스템으로부터 게이트 개방 신호를 수신하여 물리적인 게이트를 작동시킵니다.
[5] 신용카드 검증 서비스
- 설명: 시스템이 고객의 '본인 인증'을 요청할 때, 그 유효성을 검증하고 결과를 회신하는 외부 인증 시스템입니다.
[6] 상용 LLM 서비스
- 설명: 시스템이 고객의 '자연어 검색어'를 전달했을 때, 해당 텍스트의 의미와 의도를 분석하여 결과를 회신하는 외부 AI 시스템입니다.
[7] 타이머
- 설명: 시스템 외부에서 주기적인 시간 기반 이벤트를 발생시키는 주체입니다. 시스템의 배치 작업 유스케이스(설비 상태 보고 요청, 안면인식 모델 재학습)를 시작시킵니다.

### 작성가이드
- Description이 해당 QA Scenario에 대한 간략한 설명을 명확하게 기술하였는가?
- Source 및 Stimulus가 구체적인가? 즉 아키텍처 평가를 수행할 때 실제 어떤 입력이 사용되는 명확하게 판단할 수 있는가?
	- 나쁜 예) 기능의 추가
	- 좋은 예) 새로운 스케줄링 방법의 추가
- Artifact 및 Environment가 구체적인가? 즉 아키텍처 평가를 수행할 때 평가 대상인 시스템과 평가 환경을 명확하게 판단할 수 있는가? Environment는 Stimulus에 대한 Response를 관찰하며 Measure의 수치가 기대될 수 있는 구체적인 환경이 기술되어야 한다. 예) Stimulus 수신 당시 시스템의 부하
- Response가 구체적인가? 즉 해당 QA에 대한 평가를 수행할 때 Measure의 대상이 되는 동작/결과를 구체적으로 기술해야 한다.
- Response Measure가 Response와 일치하며 구체적인가? 즉 Measure의 유형 및 수치가 Response와 일치해야 하며 QA의 충족 여부 및 충족 수준을 판단할 수 있도록 Measure의 수치가 구체적인가?
- Stimulus, Environment, Response, Measure 등이 System Feature를 달성하기에 타당하고 충분한가?

### QAS-01 설비(카메라, 게이트) 고장 감지 및 실시간 알림 체계  
QA Type: Availability
Description  
지점 설비에서 고장이 발생하거나 상태 보고가 누락되면, 시스템은 신속하게 이를 감지해 지점주에게 10초 이내에 알림을 발송해야 한다.
Source of Stimulus  
지점 설비(카메라, 게이트)
Stimulus  
설비가 고장 신호를 보내거나, 일정 시간 이상(예: 30초) 상태 보고가 누락됨
Artifact  
설비 장치, 시스템, 관리자 앱
Environment  
정상 운영 중(동시 여러 설비 복수 운영 포함)
Response  
시스템이 고장/누락을 감지해 지점주에게 실시간 알림을 발송하고, 이 기록을 관리자 앱에서 확인할 수 있다.
Response Measure  
고장/누락 발생부터 알림 발송까지 P95≤15초, P99≤30초

### QAS-02 신속한 안면인식 출입 인증
QA Type: Performance
Description  
고객이나 헬퍼가 지점 출입 시, 시스템은 안면 사진을 받아 3초 이내에 출입 인증 결과를 반환하고, 성공 시 판독 성공 후 1초 이내에 게이트 개방을 실행해야 한다.
Source of Stimulus  
고객, 헬퍼
Stimulus  
출입 게이트에서 안면 사진을 등록하고 출입 인증 요청
Artifact  
지점 설비, 중앙 얼굴판독 서비스(내부 MSA)
Environment  
순간 최대 부하(Peak Load) 조건 (초당 20건의 안면 인증 요청(20 TPS)이 1분간 지속)
Response  
시스템은 안면 인증 판독과 게이트 개방 실행을 지정된 시간 내에 완료한다.
Response Measure  
- 동시 요청의 90%가 3초 이내, 99%가 5초 이내 결과 반환  
- 게이트 개방은 판독 성공 후 1초 이내

### QAS-03 자연어 검색 질의 응답의 실시간성
QA Type: Performance
Description  
고객이 앱에서 자연어로 지점 검색을 요청하면, 시스템은 피크 타임 부하 조건에서도 사용자 인지 응답 시간이 3초 이내에 완료되어야 한다.
Source of Stimulus  
고객
Stimulus  
앱에서의 자연어 기반 지점 검색 요청이 불규칙적하게 유입되며, 단위 시간(1초) 내에 최대 20건의 요청이 동시 집중 됨.
Artifact  
고객 앱, 상용 LLM 서비스, 저장소
Environment  
피크 타임(18시~20시) 부하 조건, 시간당 평균 500건의 자연어 검색 요청이 발생하는 트래픽 상황, 1,000개 이상 지점/리뷰 데이터 보유 환경
Response  
시스템이 고객의 질의 의도와 일치하는 추천 결과를 지정된 시간(3초) 내에 제공한다.
Response Measure  
전체 요청의 95% 응답 시간이 3초 이내

### QAS-04 개인정보 및 안면인식 데이터 암호화 보호 
QA Type: Security
Description  
고객의 개인정보와 안면인식 데이터를 저장하거나 전송할 때, 시스템은 강력한 암호화를 적용하여 외부 유출이나 비인가 접근으로부터 안전하게 보호해야 한다.
Source of Stimulus  
고객, 헬퍼, 관리자, 시스템 운영자
Stimulus  
개인정보와 안면인식 데이터의 저장(회원가입, 사진 등록 등), 서버 간 전송, 외부 API 연동, 조회 요청 발생
Artifact  
스마트피트니스 관리 시스템, 개인 정보 저장소, 암호화 모듈, 네트워크 API
Environment  
- 외부 클래이언트앱과 통신하는 게이트웨이만 Public Network에 위치하고 그 외 내부 시스템은 모두 Private Network
- 게이트웨이와 내부 시스템은 오직 승인된 포트/프로토콜만 허용됨
Response  
시스템은 모든 개인정보·안면인식 데이터를 저장·전송 시 암호화하며, 비인가 접근 시 데이터 복호화가 불가능해야 한다.
Response Measure  
샘플 100건의 개인정보·안면 데이터 저장 및 전송 테스트 중
- 모든 데이터가 표준 암호화(AES256, TLS 등)로 보호됨
- 외부 접근자는 복호화할 수 없음
- 암호화 누락 또는 복호화 가능 사례가 단 1건 이상 발생 시 '불충족' 판정

### QAS-06 AI 모델 교체 및 재학습의 지속적 적용성 보장
QA Type: Modifiability
Description 
시스템의 AI 모델을 교체하거나 재학습 로직을 추가·수정할 때 서비스 중단을 최소화하면서 새로운 기능을 안정적으로 적용할 수 있어야 한다. 불가피한 서비스 중단이 발생하는 경우, 영향 범위를 제한하고 빠르게 복구할 수 있어야 한다.
Source of Stimulus  
운영팀, MLOps 개발팀
Stimulus  
AI 모델 교체·업데이트, 재학습 로직 추가·수정, 버전 롤백 또는 기능 테스트 요청
Artifact  
스마트피트니스 관리 시스템, MLOps 플랫폼, 저장소, 운영 및 개발 도구
Environment  
실 운영 환경에서 실시간·배치 판독 동시 진행 중, 신규 버전 배포/테스트·롤백, 복수 모델 연동 상황
Response  
신규 AI 모델이나 재학습 로직이 서비스 중단을 최소화하면서 적용된다. 문제 발생 시 제한된 영향 범위 내에서 신속하게 복구 또는 롤백 조치가 이루어진다.
Response Measure
- 롤백 및 복구는 1분 이내에 완료한다.
- 배포/롤백 과정에서 발생하는 API 요청 실패는 전체 요청 대비 1% 미만으로 제한한다.
